# PINN-Lab web review + reorganization plan (2026-06-26)

Owner ask: full ADR-grade review of the web structure; **reorganize the App section** into scenarios +
groupings of functionalities; deep content / docs / SOTA-and-beyond. Deep-research-first → persist → validate →
implement.

## A. App reorganization (the headline) — grouped navigation

**Problem (today):** `AppPage.tsx` renders all **20 cases as one flat top-level `Tabs` strip** — unwieldy,
ungrouped, no sense of scenario or functionality. It does not scale and hides what each case DOES.

**Fix — a three-level App:**
1. **Level 1 — GROUP nav (scenarios).** Group the cases by physics **domain** (`category`): Canonical benchmarks
   (8) · Mining & mineral processing (4) · Pollution & environmental (5) · Industrial fluids & heat (2) · Control
   (1). A group tab strip + a short **group intro** (what this scenario domain is; which functionalities live here).
   (Category is clean — every case has exactly one; `system_type` is empty on the 12 static cases so it is a poorer
   grouping axis. A future "by functionality" toggle can be added once every case carries a `system_type`.)
2. **Level 2 — case cards (groupings of functionalities).** Within the active group, a **grid of case cards**; each
   card shows the case title + **functionality badges**: the **view-kit** (what visualization: Heatmap / Time-
   evolution / Trajectory / Vector-flow / UQ-band / Inverse-overlay / Spatio-temporal) and the **method** (the SOTA
   technique) and the **synthetic/real** honesty label. This makes the *groupings of functionalities* legible.
3. **Level 3 — the workbench.** Clicking a card opens the existing per-case `CaseExperiment` (Field / Live / Charts
   / Context sub-tabs) — unchanged.

**Data:** enrich `index.json` (via `build_index`/`run_all` + `contract.ts CaseIndexEntry`) with per-case
`system_type`, `view_kit`, `method`, `real_or_synthetic` so the cards render badges without loading every manifest.

**Files:** `frontend/src/pages/AppPage.tsx` (rewrite: group nav + case grid + workbench), `contract.ts`
(CaseIndexEntry + a SYSTEM_TYPE_LABELS + VIEW_KIT_LABELS map), `styles.css`, `core/manifest.py` + `pipeline.py`
(rich index entries), `i18n.ts` (group intros). Land-on-the-tool (ADR-0016) preserved: default group + first case.

## B. Deep content / docs / SOTA (follow-on, persist + implement)

- **Content depth (ADR-0016/0056/0058):** ensure every case Context is deep (problem/components/formalization+KaTeX/
  scope-honesty/how-to-read); the 6 pages referenced with real DOIs + theme-aware SVG figures; the ⓘ Architecture
  modal at full depth.
- **docs/ (ADR-0056):** a per-case `docs/cases/<case>.md` for ALL 20; method docs for every family exercised
  (incl. ode-dynamical/soft-IC, streamlines, UQ band, inverse-overlay); framework docs current.
- **SOTA + beyond-SOTA (product ladder = classical + SOTA + a NOVEL proposal):** per category, verify the SOTA is
  current (PirateNets, RBA, PI-KANs, SPINN, Hamiltonian/symplectic nets, PINO, conformal UQ) and add honest
  beyond-SOTA proposals + comparative results on the Benchmark/Experiments pages. Labelled proposals, never
  overclaimed; PINNs do not beat a good classical forward solver.

Implement A first (structural, high-value, deployable), then B in vertical slices (unit by unit, transcribed from
research, per the deep-work rule).
